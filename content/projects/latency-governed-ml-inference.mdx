This project explores how machine learning services behave under production failure conditions rather than focusing on model accuracy. The system serves predictions while enforcing latency objectives through admission control, circuit breakers, rolling performance metrics, and graceful degradation.

Instead of assuming the primary model is always available, the service continuously monitors latency and error rates, routing traffic to a lightweight fallback model when service-level objectives are violated.

<Callout type="note" title="Engineering Focus">

The primary objective was to design a resilient inference service capable of maintaining availability during overload rather than building a higher-accuracy model.

The implementation focuses on:

- Latency-aware system design
- Circuit breakers
- Admission control
- Graceful degradation
- Production observability

</Callout>

# Problem Statement

Most machine learning projects stop after deploying a model, providing little insight into how the system behaves when latency increases, resources become constrained, or failures occur. In production, these operational concerns often become more important than model accuracy.

<Callout type="info" title="Project Objective">

Design an inference service that prioritizes availability and predictable behavior under load, while making operational decisions transparent through measurable system metrics.

</Callout>

The architecture was designed around four core requirements.

<Step number={1} title="Latency SLO Enforcement">
  Continuously monitor request latency and detect SLO violations before service
  quality degrades further.
</Step>

<Step number={2} title="Graceful Degradation">
  Automatically route traffic to a fallback model when the primary service
  becomes unhealthy.
</Step>

<Step number={3} title="Admission Control">
  Limit concurrent requests to prevent resource exhaustion and uncontrolled
  queue growth.
</Step>

<Step number={4} title="Operational Visibility" isLast>
  Expose latency, error rates, circuit breaker state, and fallback usage to make
  system behaviour observable.
</Step>

# Why I Built It

Most ML portfolios demonstrate that a model can produce predictions. I wanted to explore the engineering challenges that arise after deployment, where reliability, latency, and system behaviour become the primary concerns.

<Callout type="note" title="Beyond Model Accuracy">

The project treats production failures as expected events rather than exceptional cases. Every architectural decision was made to understand how the service behaves when operating outside ideal conditions.

</Callout>

The design was driven by questions such as:

- How should an inference service respond when latency exceeds its target?
- When should requests be rejected instead of queued?
- How can degraded service remain available?
- Which metrics best reflect overall system health?
- How can overload behaviour be reproduced and analyzed?

<ArchitectureDecision title="Failure as a First-Class Design Concern">

The system assumes that latency spikes and overload will eventually occur. Instead of attempting to eliminate failures entirely, it introduces mechanisms that detect, contain, and expose them while preserving service availability.

</ArchitectureDecision>

# Engineering Decisions

<ArchitectureDecision title="Rolling Window Metrics">

Latency and error rates are computed over a rolling window, allowing routing decisions to reflect recent service behaviour rather than individual requests.

</ArchitectureDecision>

<ArchitectureDecision title="Circuit Breaker Protection">

Traffic is redirected to a fallback model when latency or error thresholds exceed predefined SLOs, reducing the impact of sustained failures.

</ArchitectureDecision>

<ArchitectureDecision title="Admission Control">

Bounded concurrency prevents uncontrolled queue growth, protecting system resources during periods of high demand.

</ArchitectureDecision>

<ArchitectureDecision title="Graceful Degradation">

The fallback model prioritizes continued availability over prediction quality, ensuring requests can still be served during degraded operation.

</ArchitectureDecision>

# Challenges & Trade-offs

A sustained load test revealed that switching to the fallback model did not immediately restore P99 latency. Queueing delays and rolling-window metrics continued to dominate end-to-end response times even after the circuit breaker activated.

This demonstrated that latency is determined by the behaviour of the entire system rather than the execution speed of an individual model.

<Callout type="warning" title="Key Observation">

Circuit breakers successfully contained failures, but they could not reverse latency already introduced by queueing and contention.

</Callout>

# Lessons Learned

- Tail latency is a system property, not a model property.
- Circuit breakers contain failures but cannot immediately recover degraded latency.
- Admission control is one of the most effective defences against overload.
- Production metrics should describe system behaviour honestly rather than present optimistic results.

<Callout type="success" title="Key Takeaway">

The most valuable outcome of this project was understanding that production ML engineering is primarily a systems engineering problem, where reliability depends on architecture rather than model accuracy alone.

</Callout>
